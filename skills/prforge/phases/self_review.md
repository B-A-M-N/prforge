## Coding Discipline Audit

Companion plugins are optional inputs. PRForge policy gates are mandatory outputs.

1. If `andrej-karpathy-skills` is installed:
   - SELF_REVIEW MUST include a mandatory discipline audit section.
   - Required questions:
     - Did I think before coding, or rush into edits?
     - Is this the smallest correct fix, or did I over-engineer?
     - Were my changes surgical, or did I rewrite unrelated code?
     - Is every changed line goal-driven and mapped to the issue?
   - Discipline audit FAILURE BLOCKS phase exit and REDIRECTS to IMPLEMENT.

2. If `andrej-karpathy-skills` is not installed:
   - SELF_REVIEW MUST use PRForge's built-in hostile review checklist with the same discipline questions.
   - Absence of the external plugin MUST NOT weaken the audit.

3. SELF_REVIEW cannot complete unless the discipline audit passes.

Use the checklist in `references/hostile-review-checklist.md`. For each "no" or
"unclear" answer, fix before proceeding.

## Key Questions

1. Does this solve the actual problem stated in the contract?
2. Did I touch only files allowed by the contract?
3. Did I break an alternate auth/provider/config path?
4. Did I preserve existing precedence chains?
5. Did I change public API signatures?
6. Are my tests meaningful (not just snapshot garbage)?
7. Would a maintainer consider this PR small and reviewable?
8. Is the PR body honest about what was validated?
9. **Is the working branch correct?** It must NOT be `main`, `master`, or any protected base branch. If on a protected branch, stop — create a feature branch and move commits before proceeding.
10. **Is all output (PR body, review responses, comments, progress notes) written in English?** Any non-English text outside an explicit translation task is a hard block — rewrite before proceeding.

## Blast Radius Review

Check `state.blast_radius` (computed automatically by the blast-radius hook):

- **Score `high`**: This is a large change. Consider whether it can be split. Verify every changed file is justified.
- **Score `medium`**: Moderate scope. Verify unexpected files are intentional and documented.
- **Score `low`**: Focused change. Good.
- **`public_api_touched == true`**: Extra scrutiny on backward compatibility.
- **`dependents_count > 5`**: Many files depend on your changes. Verify nothing is broken.

## Test Coverage Review

Check `state.blast_radius` for test coverage:

- **`tests_found_count == 0` AND `changed_files_count > 0`**: Loop back to IMPLEMENT and add tests.
  Do not surface to user — this is fixable work.
- **`unexpected_files` contains test files only**: Verify the test files actually test the changed source code, not just exist.
- **For every changed source file**: Verify the corresponding test file was updated (not just that it exists). Check `git diff` on the test file.

**If tests are missing or stale, loop back to IMPLEMENT and fix it.** Only escalate to
BLOCKED if the test infrastructure is unavailable and the gap cannot be addressed locally.

If blast radius is `high`, the approval status should be at minimum `READY_WITH_WARNINGS`.

## Guard #4: Artifact Exclusion Check

Verify `.prforge/` files are NOT tracked or staged:

```bash
# Check for staged .prforge/ files
git status --short -- .prforge/

# Check if any .prforge/ files are tracked
git ls-files .prforge/
```

If any `.prforge/` files are staged or tracked:
- Remove them from staging: `git reset HEAD .prforge/`
- If tracked, add to `.git/info/exclude` and remove from index
- Block approval until clean

Nothing screams "AI harness accident" like committing `.prforge/state.json`.

Record in `state.json` under `artifact_exclusion`.

## Guard #6: Commit Hygiene Check

Inspect all commits that will be part of the PR:

```bash
git log --format="%H %s%n%b" upstream/main..HEAD
```

Block and fix if any commit contains:
- `Co-authored-by` trailers
- `WIP`, `debug`, `temp`, `fixup`, `squash` in commit names
- `Generated by Claude`, `Generated with`, `ChatGPT`, `AI-generated` or similar bylines
- Giant AI/tool footers in commit body
- Accidental debug commits (e.g. `console.log`, `print("debug")`, `TODO: remove`)

**Fix violations by amending or squashing.** Use `git rebase -i` to clean up.
The user authored their work — the tool doesn't get a byline.

Record in `state.json` under `commit_hygiene`.
