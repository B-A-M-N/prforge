## Rule 1: Never push blindly

Always run git safety checks before any push:

```bash
git status --short
git branch -vv
git remote -v
git log --oneline --decorate -8
git diff --stat
```

Verify the branch, remote, and diff are what you expect before executing any push command.

## Rule 2: Never force-push without approval

Use `--force-with-lease`, never `--force`.

`--force-with-lease` aborts if the remote has commits you haven't fetched, protecting
against overwriting collaborator work. `--force` has no such protection.

Even with `--force-with-lease`, force-pushing is a destructive operation and
**MUST NOT happen without explicit user approval** â€” it belongs in `state.approval.approved_actions`.

## Rule 8: Never overwrite unknown user changes

If the working tree is dirty at INTAKE and the changes are not:
- Classified as existing PR work (previous PRForge run), OR
- Clearly `.prforge/` artifacts, OR
- Files the user explicitly described

Then: **stop and ask the user** before touching anything.

Do not stash and proceed. Do not assume. Ask.

Record the dirty tree classification in `state.json` before proceeding.

## Rule 9: Never co-author or attribute commits to AI

Git commit author is the configured human Git identity:
- `git config user.name` / `git config user.email`
- This is the ONLY identity used for commits

`gh api user --jq '.login'` is used for GitHub ownership/PR checks only â€” not for setting commit authorship.

**NEVER add:**
- `Co-authored-by: Claude`
- `Co-authored-by: Opus`, `Co-authored-by: Sonnet`, `Co-authored-by: Haiku`
- `Co-authored-by: Anthropic`
- Any AI model or tool name in a co-author trailer
- "Generated by Claude", "Generated with Claude Code", "AI-assisted", or any variant
- Tool attribution footers of any kind in commit messages, PR bodies, or review responses

If you find yourself writing a co-author trailer or AI footer: stop, delete it, continue.

The user authored their work â€” the tool doesn't get a byline.

## Git Disaster Recovery

If the user asks `/pr recover` or if you detect git state problems during INTAKE,
see `phases/blocked.md` for the full recovery procedure.

Quick reference â€” classify the problem first:
- Branch behind remote
- Branch diverged (local and remote have different commits)
- Wrong remote (tracking upstream instead of fork)
- Remote branch already exists (stale previous attempt)
- Local branch based on wrong base
- Detached HEAD
- Uncommitted changes that don't belong to this PR
- Accidental edits on main

Then recommend exact commands with explanation of why they are safe.
